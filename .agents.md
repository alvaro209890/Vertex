# Vertex

Standalone Vertex CLI app backed by a local DeepSeek proxy. The user-facing
product is `vertex`; `vertex-proxy` is an internal/server mode used by the app
and by advanced users who need only the Anthropic-compatible HTTP server.

Vertex was created by Alvaro Emanuel Alves Araujo. It is not an Anthropic
product. The bundled assistant must identify as Vertex and explain that it runs
DeepSeek models through the local proxy.

## Quick Start

```bash
# Dependencies: uv, Python 3.12+, Node.js 20+
cd /media/server/HD\ Backup/Servidores_NAO_MEXA/vertex
uv sync
vertex
```

On first run, `vertex` asks for email + senha (Firebase Auth). Create your
account at https://vertex-ad5da.web.app before using the CLI.

### Services

| Component | Location | Port | Description |
|-----------|----------|------|-------------|
| Express Backend | `web/backend/server.js` | 4000 | REST API (usage, auth, admin) |
| React Frontend (main) | `web/` (Firebase Hosting) | 443 | Usage dashboard |
| React Frontend (admin) | `web/admin/` (Firebase Hosting) | 443 | Admin panel |
| Cloudflare Tunnel | `cloudflared` | — | Exposes `localhost:4000` as `vertex-api.cursar.space` |
| FastAPI Proxy | `api/app.py` | 8083 | DeepSeek proxy (Anthropic-compatible) |

### Database

JSON files at `/media/server/HD Backup/Servidores_NAO_MEXA/Banco_de_dados/vertex/{uid}/`:
- `profile.json` — email, uid, created_at
- `usage.json` — `{ "models": { "model-id": { tokens: number, lastUsed: string } } }`
- `blocked.json` — `{ blocked: boolean, updatedAt: string }` (admin panel)

### Data Flow

1. User creates account at https://vertex-ad5da.web.app (RegisterPage)
2. User logs into CLI with same email + password
3. CLI validates via Firebase Auth REST API, saves token in `~/.vertex/auth.json`
4. CLI starts local proxy + vendored CLI, which sends requests to DeepSeek
5. CLI registers usage via `POST https://vertex-api.cursar.space/usage` (Bearer token)
6. Dashboard shows consolidated usage + estimated costs (USD/BRL)
7. Admin panel (https://vertex-admin-panel.web.app) controls block/unblock + metrics

## Authentication

### Firebase Auth (Email/Password)

- **Frontend**: Firebase SDK JS (`web/src/firebase.js`)
- **CLI**: REST API (`identitytoolkit`, `securetoken`) via `vertex_auth.client`
- **Backend**: Manual JWT validation with `jsonwebtoken` + Google JWKs
- **Token stored at**: `~/.vertex/auth.json` (id_token, refresh_token, expires_at, email)

The **DeepSeek API key** (`DEEPSEEK_API_KEY`) is configured server-side only
(in the project `.env`). Users never need to enter it. All accounts share the
same key through the backend + local proxy.

### Admin Auth (Simple Token)

Admin panel uses its own auth separate from Firebase:
- **Credentials**: username + password (configured via `ADMIN_USER` / `ADMIN_PASS` env vars)
- **Auth**: Bearer token with 24h expiry, stored in-memory on the backend
- **Backend**: `web/backend/routes/admin.js` — simple token-based auth middleware

### Auth Commands (CLI)

| Command | Description |
|---------|-------------|
| `vertex` | Starts CLI; if not authenticated, prompts login |
| `vertex auth login` | Interactive login (email + password) |
| `vertex auth status` | Show auth status |
| `vertex auth status --json` | JSON status output |
| `vertex auth logout` | Remove saved token |

## Models

| Model ID | Description |
|---|---|
| `deepseek/deepseek-v4-pro` | Full reasoning (thinking mode) |
| `deepseek/deepseek-v4-flash` | Fast responses (non-thinking) |

The `/model` command in the CLI shows both models. v4-pro is mapped as
Opus-tier, v4-flash as the other tiers (Sonnet, Haiku, default).

### Pricing

| Model | Input (per 1M tokens) | Output (per 1M tokens) |
|-------|----------------------|------------------------|
| deepseek/deepseek-v4-flash | $0.15 | $0.60 |
| deepseek/deepseek-v4-pro | $0.40 | $1.50 |

Cost calculation: average price = (input + output) / 2 per token. USD → BRL
conversion via `USD_TO_BRL` (default 5.0) in `web/backend/.env`.

## DeepSeek Proxy (FastAPI)

### Endpoints

- `POST /v1/messages` — Message streaming (Anthropic Messages API)
- `POST /v1/messages/count_tokens` — Token counting
- `GET /v1/models` — List available models
- `GET /health` — Health check + config fingerprint
- `GET /api/metrics` — In-memory request/token metrics

### Configuration (`.env`)

| Variable | Description | Default |
|----------|-------------|---------|
| `DEEPSEEK_API_KEY` | DeepSeek API key (required) | — |
| `MODEL` | Default model | `deepseek/deepseek-v4-flash` |
| `MODEL_OPUS` | Opus-tier override | `deepseek/deepseek-v4-pro` |
| `MODEL_SONNET` | Sonnet-tier override | `deepseek/deepseek-v4-flash` |
| `MODEL_HAIKU` | Haiku-tier override | `deepseek/deepseek-v4-flash` |
| `ANTHROPIC_AUTH_TOKEN` | Proxy auth token | `freecc` |
| `ENABLE_MODEL_THINKING` | Enable thinking blocks | `false` |
| `PORT` | Proxy server port | `8083` |
| `MESSAGING_PLATFORM` | Messaging platform | `none` |

## Auto-Start (systemd)

The backend + Cloudflare Tunnel start automatically on boot via a user systemd
service:

```bash
systemctl --user enable vertex-backend.service   # enable at boot
systemctl --user disable vertex-backend.service  # disable at boot
systemctl --user start vertex-backend.service    # start now
systemctl --user stop vertex-backend.service     # stop now
systemctl --user status vertex-backend.service   # check status
journalctl --user -u vertex-backend.service -f   # follow logs
```

Service file: `~/.config/systemd/user/vertex-backend.service`
Start script: `web/start.sh`
